# Celebrity Then-Now Segment Migration

## Status
- Deferred on purpose until after Phase 1 is implemented and reviewed.
- `SegmentDuration` should remain available as a system property and as a useful prompt-targeting input.
- The immediate Phase 1 fix removes the hidden equal-length timing behavior from `timeline/ordered` by default, but does not yet refactor `celebrity-then-now`.

## What We Learned
- The current `celebrity-then-now` blueprint avoids `NumOfSegments` and uses a `character` loop as a workaround.
- That workaround exists mainly because the blueprint needs two different intended durations:
  - meeting segments
  - transition segments
- Removing hidden video equalization makes the workaround less necessary, but it does not automatically make the blueprint easy to express as a pure `segment` loop.
- Current dimension selectors support:
  - `[loop]`
  - `[0]`
  - `[loop+1]`
  - `[loop-1]`
- Current graph syntax does **not** support richer mappings like:
  - `character = floor(segment / 2)`
  - `segment is odd/even`
  - `meeting/transition branch by segment parity`

## Recommended Next Phase
- Keep the public video flow centered on `NumOfSegments`.
- Add a small helper producer that materializes a per-segment plan from higher-level celebrity inputs.
- That helper should output canonical per-segment data such as:
  - segment kind (`meeting` or `transition`)
  - segment duration
  - start image binding
  - end image binding
  - video prompt
  - any needed narration/text metadata
- Then drive the visible video-generation pipeline from a real `segment` loop using those outputs.

## Why This Is The Best Next Step
- It keeps `NumOfSegments` as the main user-facing flow.
- It avoids adding broad graph-expression complexity too early.
- It makes mixed-duration segment blueprints easier to author without relying on hidden composer behavior.
- It keeps fail-fast behavior centralized in one place instead of scattering special cases across the blueprint.

## Alternative Options To Discuss

### Option A: Helper Producer
- Preferred default.
- Add a purpose-built producer that converts celebrity-level structure into segment-level structure.
- Lowest risk to the graph engine.

### Option B: Blueprint Restructure Without New Syntax
- Possible if the blueprint is reshaped to produce an explicit segment-plan artifact first, then loops over that artifact indirectly.
- Worth exploring if we want to stay entirely within current blueprint capabilities.
- May still end up looking like a helper-producer pattern in practice.

### Option C: Extend Graph Syntax
- Add richer selector or mapping expressions so segment-to-character relationships can be expressed directly in edges.
- Most flexible, but also the broadest and riskiest change.
- Should only be chosen if we expect several future blueprints to need the same kind of index transformation.

## Open Questions For Phase 2
- Should `NumOfSegments` become an explicit input on `celebrity-then-now`, derived from character count plus transition count, or should it be computed internally by the helper?
- Should the helper emit a single canonical `SegmentPlan[segment]` structure that all downstream producers consume?
- Do we want meeting and transition prompts generated by one director step, or should prompt generation split by segment kind once the segment plan exists?
- Are there other mixed-duration blueprints we want to support with the same pattern so we can design the helper to be reusable?
